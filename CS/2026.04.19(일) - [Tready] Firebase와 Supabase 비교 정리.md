# 2026.04.19 (일) - [Tready] Firebase vs Supabase 비교 Baas 선택 과정
> **Status**: ✅ **학습 완료**<br>**Blog**:

<br>

## 시작

Tready 프로젝트를 진행하기 앞서, 서버를 어떻게 할지 고민해야할 차례가 왔다. 나는 Spring이나 Django를 사용한 백엔드 서버를 구축하지는 못하기에, Baas 서비스를 이용해보기로 했다.

Firebase와 Supabase를 비교해서 Tready에 어떤 Baas가 어울릴지 비교해보고 판단해봤다.

<br>

## Firebase

Firebase는 **NoSQL 문서형 DB**를 사용하는 쪽이다. 데이터를 테이블처럼 딱딱 나누기보다는, 문서 단위로 저장하고 가져오는 방식에 가깝다. 그래서 처음 붙일 때는 비교적 단순하고, 인증이나 실시간 동기화 같은 기능도 바로 사용하기 편한 편이다.

<aside>

### **장점**

---

- 초기에 붙이기 **비교적 쉬운 편**이다.
- Auth, Firestore, Realtime Listener 같은 기능을 빠르게 사용할 수 있다.
- 실시간 반영이 중요한 앱에서는 시작 속도가 빠르다.
- 소규모 프로젝트에서는 무료 범위 안에서 시작하기 부담이 적다.
</aside>

<aside>

### **단점**

---

- 관계형 데이터처럼 구조를 딱 맞게 잡기는 애매한 편이다.
- 데이터끼리 연결되는 부분이 많아질수록 앱 코드에서 신경써야 할 부분이 늘어난다.
- 나중에 항목, 참여자, 담당자, 권한 같은 연결 관계가 많아지면 정리가 복잡해질 수 있다.
</aside>

Firebase는 데이터 구조가 단순하거나, 빠르게 MVP를 만들어야 할 때 잘 맞는다고 느낌.

<br>

## Supabase

Supabase는 **PostgreSQL 기반의 관계형 DB**를 사용하는 쪽이다. 테이블, PK, FK 같은 방식으로 데이터를 구조적으로 관리하는데 더 잘 맞는다. 처음에는 Firebase보다 조금 더 공부할 게 생기지만, 관계가 많은 프로젝트에서는 오히려 정리가 잘 되는 느낌이 있다.

<aside>

### 장점

---

- **관계형 데이터 구조**를 잡기에 적합하다.
- 여행, 참가자, 할 일, 담당자, 방장 권한 같은 연결을 표현하기 편하다.
- 나중에 유지보수할 때 데이터 구조를 다시 보기 좋다.
- SQL 기반이라 데이터 조회나 정리 기준이 명확하다.
</aside>

<aside>

### 단점

---

- 처음에는 **학습 비용**이 더 든다.
- SQL, 정책, 테이블 설계 같은 개념을 익혀야 한다.
- Firebase보다 처음 붙이는 속도는 조금 느릴 수 있다.

</aside>

Supabase는 데이터 간 관계가 많고, 이후에도 구조를 유지하면서 가야 하는 프로젝트에 더 맞는다고 느낌.

<br>

## Firebase / Supabase 비교

둘 다 인증과 실시간 기능은 가능하다. 
그래서 이 부분만 보면 둘 중 하나가 아예 안 맞는 수준은 아니라고 판단했다.

| 비교 항목 | Firebase | Supabase |
| --- | --- | --- |
| DB 방식 | NoSQL | 관계형 DB |
| 초반 진입 난이도 | 조금 더 쉬운 편 | 조금 더 공부 필요 |
| 관계 데이터 관리 | 직접 신경쓸 부분이 늘어날 수 있음 | 구조적으로 관리하기 좋음 |
| 실시간 기능 | 가능 | 가능 |
| 인증 기능 | 가능 | 가능 |
| 가격 부담 | 소규모에서는 부담 적음 | 소규모에서는 부담 적음 |

**가격 부분**은 처음에 가장 걱정했던 부분인데, 지금 규모에서는 둘 다 당장 무료로 시작 가능한 수준이라고 판단했다. 앱이 아직 무료이고 광고도 없고 수익도 없는 상태라서 비용이 크면 부담일 수 있었는데, 현재 예상 인원이나 사용량 기준으로는 둘 다 크게 차이가 나지 않았다. 그래서 가격은 이번 판단에서 큰 변수는 아니었고, 거의 **동점**으로 봤다.

실시간 동기화도 Tready에서는 중요한 기능인데, 이 부분 역시 둘 다 가능하다고 봤다. 결국 남는 건 데이터 구조를 어떻게 가져갈지가 더 중요했다.

<br>

## Tready 기준으로 생각

Tready는 여행 하나 안에서도 참가자, 공용 할 일, 개인 할 일, 담당자, 권한 같은 식으로 **서로 연결되는 데이터**가 계속 생긴다. 단순히 목록 몇 개 저장하는 수준보다는, 항목들끼리 관계를 정리해가야 하는 쪽에 더 가깝다고 봤다.

그래서 처음엔 Firebase가 더 쉬워 보이긴 했지만, 나중에 관계를 다시 정리하거나 유지보수할 때는 **Supabase 쪽이 더 적합하다고 판단**했다.

정보처리기사 공부할 때 PK, 외래키, SQL 같은 개념을 잠깐이라도 본 적이 있어서, 아예 처음부터 아무것도 모르는 상태는 아니라고 생각했다. 다시 공부하면 충분히 따라갈 수 있을 것 같았고, 이 부분은 학습 비용이 있더라도 감당 가능한 범위라고 봤다.

<br>

## 결론

이번 Tready 프로젝트에서는 **Supabase**가 더 적합하다고 판단했다.

이유는 단순하다. Firebase가 초반 진입은 더 쉬울 수 있지만, Tready는 관계가 있는 데이터가 계속 생기는 구조라서 이후 정리와 유지보수까지 생각하면 Supabase 쪽이 더 잘 맞는다고 느꼈다.

가격도 처음에는 걱정했지만, 지금 예상하는 사용자 수와 사용량 기준으로는 둘 다 당장 무료로 시작 가능한 수준이라 큰 차이는 아니었다. 실시간 기능도 둘 다 가능하다고 봤기 때문에, 최종적으로는 **데이터 구조와 이후 유지보수성** 쪽에서 Supabase 쪽에 더 점수를 줬다.
