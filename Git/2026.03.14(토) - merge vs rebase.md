# 2026.03.14(토) - merge vs rebase
> **Status**: ✅ **학습 완료**<br>**Blog**:

<br>

git에서 merge와 rebase는 최종 코드 결과물만 보면 동일하다. 

타겟 브랜치의 변경사항을 내 브랜치에 반영하고 싶을 때 쓰는건데, 서로 동작 방식이랑 히스토리 그래프가 달라진다.

## merge

merge는 두 브랜치의 공통 조상(base commit)을 기준으로 3-way merge를 진행한다.

- **3-way merge: Git이 두 브랜치를 합칠 때 사용하는 기본 알고리즘.**

충돌이 있으면 한 번에 전부 해소하고, 그 결과로 **merge commit** 이 하나 생긴다.

```swift
// feature 브랜치를 main에 merge
// 공통 조상: C

main:    A --- B --- C ------- M  (merge commit)
                      \       /
feature:               D --- E
```

→ 히스토리에 merge commit(M)이 남고, **기존 커밋들의 해시값은 그대로 유지**된다. 

브랜치 분기 흔적이 그래프에 그대로 남아있음.

<br>

## rebase

rebase는 merge랑 달리, 내가 작업한 커밋들을 일단 떼어내고 **타겟 브랜치의 HEAD(브랜치명: C)를 새로운 시작점으로 해서 그 위에 하나씩 다시 붙인다.**

그래서 커밋을 붙일 때마다 충돌이 생기면 그때그때 해소해야 한다.

```
# feature 브랜치를 main 기준으로 rebase

[before]
main:    A --- B --- C
                \
feature:         D --- E

[after]
main:    A --- B --- C
                      \
feature:               D' --- E'
```

→ D, E 커밋이 **D', E'** 로 바뀐다. 내용은 같지만 **해시값이 달라진 새로운 커밋**이다.

분기 흔적 없이 **선형(linear) 히스토리로 깔끔하게 유지**되고, merge commit도 없어서 커밋이 불필요하게 늘어나지 않는다.

## 언제 무엇을 써야할까

---

개인 프로젝트를 할 때는 rebase를 써도 전혀 문제가 없다. 오히려 히스토리가 깔끔해져서 좋다.

하지만 협업에서 이미 공개된 브랜치에 rebase를 하면 문제가 생긴다. 

내쪽에서 해식밧이 바뀐 커밋들이 생기는데, 다른 팀원의 local에서는 원래 해시값의 커밋이 남아있기 때문에, 히스토리가 서로 달라진다.

결국 합치게 될 경우에 **강제로 push(`-- force`)** 를 써야하는 상황이 발생한다.

```swift
// 공개된 브랜치에서 rebase 하면

[팀원 A (나)]
feature: A --- B --- C'--- D'   ← rebase 후 해시가 바뀜

[팀원 B]
feature: A --- B --- C --- D    ← 아직 원래 해시 그대로

// 이 상태에서 push하면 충돌. force push 해야함
// → 팀원 B의 작업이 덮어씌워질 수 있는 위험
```

<br>

## 사용 방법 정리

- **PR 올리기 전, 로컬에서 작업 중** → `rebase` 써서 최신 main 반영하고 히스토리 깔끔하게
- **PR 머지할 때** → `merge` 써서 merge commit 남기고 이력 보존

즉, **공개되기 전엔 rebase, 공개된 이후엔 merge** 가 기본. 다만 개인프로젝트는 혼자 쓰기때문에 rebase로 선형 관리를 해도 문제가 없다.
